# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - Data Modeling
    - 데이터 모델링 실습
    - 논리 데이터를 까마귀발 표기로 그릴 때 관계를 양방향으로 읽을 수 있도록 학습
  - Step 1 과제
    - HTTP Request에서 주요 정보만 파싱하여 로깅하도록 수정
    - Thread를 사용하도록 되어있던 코드를 concurrent 패키지를 사용하도록 수정(ThreadPool)
    - 기존의 RequestHandler에 몰려있던 책임을 분리하기 위해 HttpRequest, HttpRequestParser, StaticResourceHandler 클래스 생성
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - Step 1에 올라온 Codex의 PR 리뷰 수정하기(예외 처리, 보안적 요소, 등)
  - Step 2 내용 구현하기

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 책임 분리를 위해 HttpRequest, HttpRequestParser, StaticResourceHandler 클래스를 구현한 것.
- 그렇게 판단한 이유: 자바를 코딩 테스트를 위해 사용하다보니 객체지향 프로그래밍에 대한 걸 많이 잊은 채 하나의 클래스에 막 코드를 짠 듯 해서 확장성을 위해 클래스를 나눠봤다.
- 고려했지만 선택하지 않은 대안: HttpRequest 클래스만 만든 뒤 기존 RequestHandler 클래스에 메서드로 나머지 기능 구현하기
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 당장 봤을 때에는 코드량이 늘어났지만, 추후 기능을 추가하거나 확장할 때 HttpRequest에 관한 부분과 정적 파일에 관한 부분이 분리되어 적은 코드의 수정으로 확장이 가능해졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: ERD 작성 중 주문 - 결제 - 결제수단의 관계의 기수성을 파악하는데 어려움이 있었다.
- 처음 추측한 원인: 주문 1개당 한 번의 결제가 묶인다고만 생각했다.
- 원인을 확인하기 위해 시도한 방법: 예제 데이터를 AI로 뽑아서 표에 채워봤다.
- 실제 원인: 주문 1개에는 한 번의 결제로 보일 수 있지만, 여러 개의 결제 방식(카드, 쿠폰, 포인트)를 사용해서 결제하는 경우가 있었기에 주문과 결제는 1:N의 관계였다.
- 해결 방법: 위에 서술된 결론이 도출되었고, 생각보다 다른 요소들이 더 있을 수도 있겠다고 생각한다.(결제 내역을 추가한다거나 하는 등.)
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 다음에도 예제 데이터를 뽑아보고 표에 넣어서 확인해볼 것 같다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 오늘은 학습에 집중해서 크게 개발과 관련한 쪽에서 잘한 행동이라고 할 것이 없는 것 같다. 내일 좀 더 개발에 비중을 둬보자.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 막힌 부분이 생겼을 때 지피티에게 바로 정답을 요구했던 거 같다.
- 그렇게 된 원인: 최근에 짧은 기간의 프로젝트를 하면서 잦은 바이브 코딩을 하다보니 의존성이 좀 생겼다.
- 다음에는 다르게 해볼 구체적인 행동: AI는 이제 뗄 수 없는 존재라고 생각한다. 하지만 학습을 위해서는 정답이 아닌 힌트 정도만 받은 뒤 직접 코드 작성에 대한 부분이나 자바의 이론적인 부분을 공부해서 해결하려고 한다.
