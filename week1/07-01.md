###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: io 패키지를 사용하여 파일을 읽는 readFile 메서드를 구현한 것.
- 그렇게 판단한 이유: nio 패키지를 사용하지 않는다는 요구사항이 있던 것이 처음이었지만, 이후 nio 패키지를 사용해도 괜찮다는 것을 알았음에도 실제 내부 동작이 이렇게 이루어진다는 것을 학습하게 되었다.
- 고려했지만 선택하지 않은 대안: nio 패키지를 사용하느냐 마느냐 말고는 다르게 고려한 내용은 없었습니다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: io 패키지만으로 직접 구현하여 저수준 이해엔 좋지만 코드량이 많아지고, 오류 처리를 하지 않은 코드가 되었으며, 테스트가 어려워졌다(파일시스템 직접 의존). 실제 코드를 작성한다면 Files.readAllBytes(Path)로 대체 + url 경로 검증(path traversal 방지)을 추가할 것 같다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: index.html을 불러왔을 때 svg, png, 등의 static 파일들이 불러와지지 않았었다.
- 처음 추측한 원인: 처음엔 index.html만을 불러오도록 작성해서 그 외의 static 파일 요청을 처리하지 못한다고 생각하여 코드를 범용성있게 수정했다.
- 원인을 확인하기 위해 시도한 방법: logger로 요청을 출력하도록 해놓긴 했지만, chrome의 개발자 도구를 사용해서 패킷 내용을 확인했다.
- 실제 원인: step2에 서술되어있는 MIME 처리 즉, Content-Type 처리가 html로만 되어있어 불러오지 못했다.
- 해결 방법: 아직 해결하진 않았지만, content-type에 파일의 타입에 맞게 처리하도록 코드를 작성해볼 것 같다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: chrome을 자주 쓰고 요새는 AI에 의존해서 개발을 하다보니 코드를 잘 보지 않으면서 동시에 로깅도 잘 하지 않게 된 것 같다. 다음부턴 내가 확인하기 위해 만들어둔 로그를 먼저 확인할 것 같다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 2번에서 서술했듯이, 이번 자바 과제는 여타 패키지, 프레임워크를 사용하지 않고 구현하는 것이 목표라고 생각한다. 이처럼 이번 과제를 진행할 때에는 최대한 java의 기본 패키지를 사용해서 필요한 메서드를 구현할 예정이고 오늘도 그렇게 구현했다.
- 그 접근이 도움이 된 이유: 코드의 동작이 기억이 남으려면 내부 코드를 살펴보거나, 직접 그 메서드를 구현해보는 것이 좋다고 생각하고, 실제로 나에겐 그런 학습 방식이 맞는 것 같다.
- 다음에도 반복하고 싶은 점: 이후의 과제에서도 편히 쓸 수 있는 패키지가 있다고 하더라도, 바로 적용하는 것이 아닌 직접 메서드를 구현해보고 어떤 부분을 놓쳤는지, 어떤 부분이 부족한지 비교해보며 메서드를 사용할 것 같다.
