### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:  
  앞으로 `GET` 외의 HTTP method도 처리해야 할 것 같아서, method별 처리를 맡는 `Mapper` 클래스를 따로 두는 방향을 생각했다.

- 그렇게 판단한 이유:  
  처음에는 `GET`으로 정적 파일만 내려주면 된다고 봤다. 그런데 회원가입처럼 요청 목적이 달라지는 흐름을 보니, `RequestHandler` 안에 계속 조건문을 추가하면 금방 복잡해질 것 같았다.

- 고려했지만 선택하지 않은 대안:  
  `RequestHandler`에서 `if`나 `switch`로 method를 직접 분기하는 방법도 생각했다. 지금은 빠르게 구현할 수 있지만, 기능이 늘어나면 한 클래스에 책임이 몰릴 가능성이 크다고 봤다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:  
  아직 구현하진 못했지만, 앞으로 요청 처리 구조를 나눌 기준이 생겼다. method별 책임이 분리되면 테스트 범위도 더 명확해지고, 새 기능을 추가할 때 기존 정적 파일 처리 코드를 덜 건드릴 수 있을 것 같다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:  
  디렉토리 경로로 요청이 들어왔을 때 `index.html`을 읽어야 했는데, 이때 `Content-Type`을 어떻게 처리할지에서 막혔다. 타입이 제대로 잡히지 않으면 브라우저에서 페이지가 열리지 않고 파일이 다운로드됐다.

- 처음 추측한 원인:  
  처음에는 `/`만 `/index.html`로 바꾸고 있어서 생긴 문제라고 생각했다. 그래서 다른 디렉토리 요청도 비슷하게 처리하면 된다고 봤다.

- 원인을 확인하기 위해 시도한 방법:  
  요청 path를 정규화하는 흐름, 실제 파일을 찾는 흐름, `Content-Type`을 결정하는 위치를 따로 확인했다.

- 실제 원인:  
  실제로 읽을 파일은 `StaticResourceHandler`가 알고 있는데, 타입 판단은 `RequestHandler` 쪽에 있어 처리 흐름이 어긋났다. 디렉토리 요청이 `index.html`로 바뀌는 경우를 자연스럽게 반영하기 어려웠다.

- 해결 방법:  
  디렉토리 요청이면 해당 디렉토리의 `index.html`을 읽도록 `StaticResourceHandler`에서 처리했다. 그리고 `Content-Type`을 판단하는 메서드도 `StaticResourceHandler`로 옮겨 정적 파일 처리 책임을 한곳에 모았다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:  
  요청 path와 실제로 서빙할 파일 path가 같은지 먼저 확인할 것이다. 다음에는 문자열 확장자 비교를 늘리기보다, enum으로 파일 타입과 `Content-Type`을 묶는 방식도 고려해보고 싶다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:  
  구현하기 전에 구조를 계속 AI agent에게 확인받았다. 백엔드 구조를 처음부터 직접 짜본 경험이 많지 않아서, 책임을 어떻게 나눌지 계속 고민해봤다.

- 그 접근이 도움이 된 이유:  
  직접 짜보니 기능이 조금만 늘어나도 수정할 부분이 계속 생겼다. `HttpResponse` 분리, 정적 리소스 처리, query parameter 처리 등을 하면서 각 클래스의 역할을 더 의식하게 됐다.

- 다음에도 반복하고 싶은 점:  
  다음에도 구현 전에 클래스의 역할을 먼저 말로 정리해볼 것 같다. AI에게도 전체 구현을 맡기기보다, 내가 생각한 구조가 괜찮은지 확인받는 식으로 활용하고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:  
  하다가 답답해서 AI에게 그냥 해달라고 맡긴 부분이 있었다. 원래는 `HttpRequest`를 `record`로 리팩토링하는 정도였는데, AI가 전체 코드를 크게 바꿔버렸다. 원복하는 과정에서 내가 고민해서 짠 코드도 같이 날아갔다.

- 그렇게 된 원인:  
  수정 범위와 제약 조건을 명확하게 주지 않았다. 어떤 파일만 바꿀지, 기존 동작은 유지할지, 건드리면 안 되는 로직은 무엇인지 구체적으로 말하지 않았다.

- 다음에는 다르게 해볼 구체적인 행동:  
  다음부터는 "이 파일만 수정", "동작 유지", "컴파일에 필요한 최소 변경만"처럼 조건을 먼저 걸 것이다. 큰 변경 전에는 현재 상태를 커밋하거나 diff를 확인해서, 원복하더라도 내 코드가 같이 사라지지 않게 할 것이다.



# 주간 프롬프트 회고

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트: 
```
기존 구조에선 path가 "/"일 때만 /index.html로 변환해서 타 폴더에 접근 할 때는 index.html을 참고하지 않는 문제가 있어. 그래서 다음과 같은 로직으로 수정하려고 하는데 어떤 거 같아?
1. path의 파일이 directory인지 확인
2. 기존의 content type을 StaticResourcesHandler의 메서드로 옮김
3. content type을 구할 때 Path의 파일 타입을 확인하게 해서 디렉토리면 html을 return 하도록 수정
```

선택한 이유: 학습을 위한 방향으로 AI를 사용하고 있기 때문에 내가 생각한 구조가 맞는지 틀렸는지, 그리고 틀렸다면 왜 틀렸고 어떤 구조를 사용해야 하는지 알아내는 목적으로 사용했기 때문에 선택했습니다.

### 2. 원본 프롬프트 구조 분석
- 문제 상황: path가 디렉토리일 때 index.html을 리턴하지 않는 문제가 있었습니다.
- 원하는 결과를 어떻게 요구했는가: 변경을 요청하거나 답을 출력해달라고 하지 않고, 내가 생각한 로직이 맞는지에 대한 판단을 맡겼습니다.
- 필요한 맥락이 포함되어 있는가: 내가 어떤 문제 상황이고, 무엇을 하려고 하고, 무엇을 원하는지에 대한 작성은 한 것 같지만, 전체적인 코드에 대한 설명은 부족했던 것 같습니다.
- 판단 기준이나 제약 조건이 있는가: 크게 판단 기준이나 제약 조건을 걸지 않았습니다.
- 출력 방식이 명확한가: 출력 방식에 대해서 서술하진 않았습니다.
- 나의 가설, 우려, 의도가 드러나는가: 제 의견을 가설로 제시하고 의도도 제시하였지만, 우려 사항에 대해서는 작성하지 않았습니다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
현재 StaticResourcesHandler는 요청 path가 "/"인 경우에만 "/index.html"로 변환한다.
그래서 "/docs", "/admin/"처럼 특정 디렉토리로 접근하면 해당 디렉토리의 index.html을 서빙하지 못하는 문제가 있다.

다음 요구사항에 맞게 기존 구조를 최소 변경으로 수정해줘.

요구사항:
1. 요청 path가 디렉토리를 가리키는 경우, 해당 디렉토리 내부의 index.html을 서
빙한다.
    - "/" -> "/index.html"
    - "/docs" -> "/docs/index.html"
    - "/docs/" -> "/docs/index.html"
2. 일반 파일 요청은 기존처럼 동작해야 한다.
    - "/main.css"는 css 파일로 서빙
    - "/app.js"는 js 파일로 서빙
3. content type 결정 로직은 StaticResourcesHandler 내부 메서드로 분리한다.
4. content type은 최종적으로 서빙할 실제 파일 경로 기준으로 판단한다.
    - 디렉토리 index.html 응답은 text/html이어야 한다.
5. 존재하지 않는 path, 파일이 아닌 path, path traversal 공격("../")에 대한 기존 방어 로직이 있다면 유지한다.
6. 관련 테스트가 있으면 추가하거나 수정해줘.

수정 전에는 현재 path resolve 흐름을 먼저 확인하고, 리소스 경로 결정과 content type 결정의 책임이 섞이지 않도록 구현해줘.
```

### 4. AI가 개선한 프롬프트 분석
- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등
```
원본 프롬프트에 비해 조금 더 상세하게 코드의 위치 설명이나 문제 상황을 설명했고, 요구사항에 대해서도 말로만 하는게 아닌 실제 예시를 작성하여 수정하더라도 목적에 맞게 수정하도록 작성했다.

어떤 파일을 어떻게 분리하고, 어떤 로직은 유지해야 하는지를 명시하여서 AI가 임의로 작성한 코드더라도 문제가 발생하지 않도록 제약을 걸어주었다.

출력 형식에 대한 건 없지만 코드를 작성해달라는 명시를 했기 때문에 실제 파일을 수정할 것으로 보인다.

위의 이유를 바탕으로 실제 결과를 출력 했을 때 내가 입력한 프롬프트보다 좀 더 내 기존 코드에 알맞는 코드를 작성해줄 것으로 보인다.
```
