# be-DailyJob 

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:
  - 요청의 `Cookie` 헤더에서 원하는 쿠키 값을 꺼낼 수 있도록 `HttpRequest.getCookie(name)`을 추가했다. 이제 로그인 이후 브라우저가 보내는 `Cookie: sid=...` 값을 서버에서 다시 읽을 수 있다.
  - 로그아웃 라우트를 추가했다. `POST /user/logout` 요청이 들어오면 쿠키의 `sid`를 기준으로 세션을 삭제하고, 응답에는 만료 쿠키를 내려주도록 만들었다.
  - 로그인/로그아웃 책임을 `UserController`에서 `AuthController`로 분리했다. 회원가입은 사용자 생성 책임으로 남기고, 인증 흐름은 별도 컨트롤러에서 처리하게 했다.
  - 요청 처리와 인증 흐름에 필요한 로그를 추가했다. 라우트 매칭, 로그인 성공/실패, 로그아웃, 세션 생성/삭제처럼 흐름을 따라가야 하는 지점에 로그를 남겼다.
  - 예외가 날 수 있는 부분 중 단순 방어로 막을 수 있는 곳을 보완했다. `HttpResponse`는 null body나 null headers를 받아도 기본값으로 처리하게 했고, 서버 포트 인자 파싱도 방어했다.
  - 로그인과 쿠키 처리 테스트를 추가했다. 로그인 성공 시 `Set-Cookie`가 내려가는지, 로그인 실패 시 `401`이 내려가는지, 로그아웃 시 세션이 삭제되고 쿠키가 만료되는지 확인했다.
  - Step5 구현 내용을 `docs/step5.md`로 문서화했고, PR 본문 초안도 따로 정리했다.
  - Step5 리팩토링 이후에도 남아 있는 객체지향 설계 문제를 `docs/step5-oop-review.md`에 정리했다. 현재 클래스 의존성과 지향해야 할 의존성을 Mermaid로 그려 비교했다.
  - Step7에서 처리할 에러/보안/라우팅 개선 후보를 `docs/step7-error-handling-plan.md`에 정리했다. 특히 없는 경로의 비-GET 요청이 `405`로 내려가는 문제를 Step7 개선 항목으로 남겼다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - `404 Not Found`와 `405 Method Not Allowed`를 구조적으로 구분하는 작업은 아직 하지 않았다. 라우터와 정적 리소스 처리 책임을 다시 나눠야 해서 Step7에서 처리하기로 했다.
  - 세션 만료, `HttpOnly`, `SameSite`, 비밀번호 해싱, CSRF 같은 인증 보안 항목은 아직 구현하지 않았다. Step7 이후 보안 개선 후보로 남겨뒀다.
  - `Database`, `SessionManager` 같은 static 저장소 의존을 줄이는 작업도 아직 남아 있다. 객체지향 리팩토링 때 같이 다룰 예정이다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  - `UserController`에 몰려 있던 인증 관련 책임을 `AuthController`로 분리했다. 회원가입은 `UserController`에 남기고, 로그인과 로그아웃은 인증 흐름으로 보고 별도 컨트롤러에서 처리하게 했다.
- 그렇게 판단한 이유:
  - 처음에는 회원가입, 로그인, 로그아웃이 모두 사용자와 관련된 기능이라 `UserController`에 있어도 된다고 생각했다. 그런데 구현이 진행될수록 성격이 다르다는 점이 보였다.
  - 회원가입은 사용자를 생성하는 기능이고, 로그인과 로그아웃은 이미 존재하는 사용자를 기준으로 세션을 만들거나 제거하는 기능이다. 둘을 한 컨트롤러에 계속 두면 `UserController`가 사용자 생성, 인증, 세션, 쿠키 응답까지 모두 알게 된다.
  - 특히 로그인은 `Database` 조회, 비밀번호 검증, `SessionManager` 호출, `Set-Cookie` 응답 생성까지 이어진다. 이 흐름을 회원가입 컨트롤러에 계속 붙여두면 이후 세션 만료나 쿠키 보안 속성을 추가할 때 책임이 더 섞일 것 같았다.
- 고려했지만 선택하지 않은 대안:
  - `UserController` 안에 `create`, `login`, `logout`을 모두 두는 방법도 가능했다. 과제 규모만 보면 가장 빠른 방식이다.
  - 하지만 기능이 늘어날수록 컨트롤러 이름과 실제 책임이 어긋날 가능성이 컸다. 그래서 이번 단계에서는 `AuthController`를 따로 두는 쪽이 더 낫다고 판단했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 회원가입과 인증 흐름을 구분해서 볼 수 있게 됐다. 라우터에서도 `/user/create`는 `UserController`, `/user/login`과 `/user/logout`은 `AuthController`로 연결되니 흐름이 더 읽힌다.
  - 테스트를 볼 때도 회원가입 테스트와 로그인/로그아웃 테스트의 관심사가 나뉜다.
  - 아직 완전히 객체지향적인 구조라고 보기는 어렵지만, 적어도 한 컨트롤러에 모든 사용자 관련 흐름을 몰아넣는 상태에서는 벗어났다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  - Step5 리팩토링을 어느 정도 마치고 보니, 파일은 많이 나눴지만 객체지향적인 구조라고 말하기에는 아쉬운 부분이 많았다. `AuthController`를 분리했는데도 여전히 컨트롤러가 저장소, 세션, 쿠키 문자열, 응답 생성까지 많이 알고 있었다.
- 처음 추측한 원인:
  - 처음에는 클래스를 더 나누면 자연스럽게 객체지향적인 구조가 될 거라고 생각했다. `controller`, `router`, `resource`, `session`처럼 패키지를 나누고 나니 구조가 꽤 정리된 것처럼 보였다.
- 원인을 확인하기 위해 시도한 방법:
  - 코드를 다시 보면서 각 클래스가 무엇을 알고 있는지 확인했다. 특히 `AuthController`, `RequestRouter`, `Database`, `SessionManager`, `HttpResponse` 사이의 의존성을 봤다.
  - 오브젝트라는 책을 읽으면서, 단순히 클래스를 나누는 것과 객체가 자기 책임을 갖는 것은 다르다는 점을 다시 생각하게 됐다.
  - Codex의 도움을 받아 현재 클래스 의존성과 지향해야 할 의존성을 Mermaid로 그려봤다.
- 실제 원인:
  - 지금 구조는 책임을 나누기 시작한 단계에 가깝다. 아직 컨트롤러가 서비스 역할까지 하고 있고, `Database`와 `SessionManager` 같은 static 저장소에 직접 의존한다.
  - 쿠키 문자열도 컨트롤러에서 직접 만들고 있다. `HttpRequest`와 `HttpResponse`도 내부 구현을 꽤 많이 드러낸다.
  - 결국 문제는 파일 수가 아니라 의존성 방향과 책임의 위치였다.
- 해결 방법:
  - 아직 해결하지 못했다. 오늘은 바로 코드를 다시 뜯어고치기보다, 내가 지향해야 할 구조를 문서로 정리하는 데 그쳤다.
  - `docs/step5-oop-review.md`에 현재 의존성과 지향 의존성을 정리했고, 다음 리팩토링 후보로 `SessionCookie`, `AuthService`, `HttpRequest` 캡슐화, static 의존 제거를 남겼다.
  - Step7에서 에러 처리와 라우팅 책임을 다시 만질 예정이라, 그때 OOP 관점의 리팩토링도 같이 진행할 계획이다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 클래스를 나눴다는 사실만 보고 구조가 좋아졌다고 판단하지 않는다.
  - 각 객체가 무엇을 알고 있는지, 누구를 직접 호출하는지, 변경이 생기면 어느 클래스가 같이 흔들리는지 먼저 본다.
  - 리팩토링을 시작하기 전에 현재 의존성과 목표 의존성을 간단히 그려본다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
  - Step5 기능과 큰 리팩토링이 어느 정도 끝난 뒤, 기본적인 예외 처리와 로그를 한 번 정리했다.
  - `HttpResponse`가 null body나 null headers를 받아도 바로 실패하지 않도록 방어했고, 서버 실행 인자로 들어오는 포트 값도 잘못된 경우를 고려했다.
  - 요청 처리 흐름, 라우팅, 로그인 성공/실패, 로그아웃, 세션 생성/삭제처럼 디버깅할 때 확인해야 할 지점에 로그를 남겼다.
- 그 접근이 도움이 된 이유:
  - 기능 구현 중에는 정상 흐름에 집중하게 된다. 그런데 서버는 요청이 잘못 들어오거나 값이 비어 있는 상황도 계속 만난다. 예외 처리와 로그를 정리해두니 문제가 생겼을 때 어디서 흐름이 끊겼는지 보기 쉬워졌다.
  - 특히 로그인과 로그아웃은 브라우저, 쿠키, 서버 세션이 같이 움직인다. 로그가 없으면 `Set-Cookie`가 내려갔는지, 다음 요청에서 `Cookie`가 들어왔는지, 세션이 실제로 삭제됐는지 확인하기가 번거롭다.
  - 테스트로 확인한 내용과 로그로 볼 수 있는 흐름이 맞물리면서, 구현이 단순히 동작하는 수준을 조금 벗어났다고 느꼈다.
- 다음에도 반복하고 싶은 점:
  - 기능을 붙인 뒤에는 정상 동작만 확인하지 말고, 예외가 날 만한 입력과 로그로 추적해야 할 지점을 같이 본다.
  - 로그는 아무 데나 많이 찍기보다 요청 흐름을 따라가는 데 필요한 지점에 남긴다.
  - 리팩토링이 끝난 것처럼 보이는 시점에 한 번 더 예외 처리, 로그, 테스트를 확인한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
  - Step5 리팩토링을 많이 했지만, 처음부터 객체의 책임과 의존성 방향을 충분히 그려놓고 시작하지는 못했다. 구현이 어느 정도 끝난 뒤에야 `AuthController`가 너무 많은 책임을 갖고 있다는 점이 더 잘 보였다.
  - `Database`와 `SessionManager`를 static으로 직접 호출하는 구조가 테스트와 확장에 불리하다는 점을 알면서도, 이번 단계에서는 그대로 두었다.
  - `404`와 `405` 구분 문제도 라우터를 만들 때 미리 더 엄밀하게 생각했으면 좋았을 부분이다.
- 그렇게 된 원인:
  - 우선 Step5 요구사항을 끝내는 데 집중하면서, 동작하는 구조와 객체가 자기 책임만 갖는 구조를 충분히 분리해서 보지 못했다.
  - 과제 규모가 작다 보니 static 저장소나 문자열 쿠키 조립이 당장은 편하게 느껴졌다. 하지만 기능이 늘어나니 그 편한 선택들이 의존성으로 남았다.
  - HTTP 상태 코드도 등록된 동적 route만 기준으로 생각했고, 정적 리소스 존재 여부까지 포함한 전체 라우팅 기준은 뒤늦게 고민했다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 기능을 붙이기 전에 요청 흐름뿐 아니라 객체 간 의존성도 간단히 그려본다.
  - 컨트롤러가 저장소, 세션, 쿠키, 응답 생성을 모두 알고 있으면 바로 서비스나 전용 객체로 나눌 후보인지 확인한다.
  - 라우터를 수정할 때는 정확히 일치하는 route, path는 있지만 method가 틀린 경우, 정적 리소스 존재 여부, 완전히 없는 경로를 구분해서 본다.
