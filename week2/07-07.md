# be-DailyJob

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:
  - step5를 진행하기 전에 웹 서버 패키지 구조를 먼저 정리했다. `controller`, `http`, `router`, `resource`, `session`으로 역할을 나누고, 기존 `HandlerMapping`은 `RequestRouter`로 이름을 바꿨다.
  - `RouteHandler`, `RouteKey`, `Routes`를 기준으로 method와 path 조합을 라우팅하도록 정리했다. `/user/create`, `/user/login` 같은 경로 문자열은 `Routes`에 모았다.
  - 정적 리소스 처리 책임을 나눴다. `StaticResourceHandler`는 요청을 응답으로 바꾸는 흐름만 맡고, 실제 파일 해석은 `StaticResourceResolver`, 파일 읽기는 `StaticResourceReader`, 파일과 Content-Type 정보는 `StaticResource`가 맡도록 했다.
  - 변수명과 포맷을 정리했다. `in`, `out`, `br`처럼 짧게 줄인 이름을 `inputStream`, `outputStream`, `reader`처럼 역할이 보이게 바꿨고, `request`, `httpRequest`, `path`, `requestPath` 같은 이름도 쓰임에 맞게 맞췄다.
  - 로그인 요청 경로를 추가하고, 로그인 폼이 `POST /user/login`으로 요청을 보내도록 연결했다.
  - `Session`, `SessionManager`를 추가해 로그인 성공 시 세션을 생성하고, 세션 ID를 서버 메모리에 저장할 수 있게 했다.
  - 로그인 성공 응답은 `200 OK + Set-Cookie` 방식으로 구현했다. 서버가 직접 redirect하지 않고, 프론트에서 성공 응답을 받은 뒤 `/index.html`로 이동하도록 했다.
  - 로그인 실패 시 `401 Unauthorized`를 반환하고, 로그인 화면 아래에 실패 메시지를 보여주도록 했다. 실패하면 비밀번호 입력값도 비우도록 했다.
  - `HttpResponse`의 `Location`, `Set-Cookie` 개별 필드를 `Map<String, String> headers`로 통합했다. 앞으로 응답 헤더가 늘어나도 필드를 계속 추가하지 않아도 되는 구조로 바꿨다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 요청의 `Cookie` 헤더를 파싱해서 `sid`를 꺼내는 기능은 아직 남아 있다.
  - 로그아웃 라우트와 세션 삭제 처리는 아직 구현하지 않았다.
  - 로그인 상태에 따라 메인 화면의 UI를 다르게 보여주는 작업은 아직 하지 않았다.
  - `HttpResponse`를 `headers Map`으로 바꾸는 과정에서 세부 값이 정확히 들어가는지 한 번 더 확인해야 한다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  - 로그인 성공을 `302 redirect`가 아니라 `200 OK + Set-Cookie`로 처리하고, 화면 이동은 프론트에서 하기로 했다.
- 그렇게 판단한 이유:
  - form 기본 제출 방식이면 서버가 `302`를 내려 이동시키는 흐름이 자연스럽다. 하지만 로그인 실패를 같은 화면에서 보여주려면 프론트가 응답 상태를 직접 확인해야 했다.
  - `fetch`로 로그인 요청을 보내면 성공과 실패를 화면에서 직접 제어할 수 있다. 성공하면 `/index.html`로 이동하고, 실패하면 현재 로그인 화면에 메시지를 보여주면 된다.
  - 이 방식에서는 실패 응답으로 `401 Unauthorized`를 쓰는 것도 더 자연스러웠다. 서버는 인증 실패라는 의미를 상태 코드로 알려주고, 프론트는 그 상태를 보고 화면만 바꾸면 된다.
- 고려했지만 선택하지 않은 대안:
  - 로그인 성공 시 `302 Found + Location + Set-Cookie`를 내려 브라우저가 바로 이동하게 하는 방식도 생각했다. 이전 step에서 redirect를 구현했기 때문에 처음에는 이쪽이 더 익숙했다.
  - 하지만 이 방식은 실패 메시지를 현재 화면 안에 보여주려면 애매했다. 브라우저 기본 form submit은 실패 응답을 현재 페이지의 일부로 다루기보다 응답 페이지 자체로 보여줄 수 있기 때문이다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 성공과 실패 흐름이 HTTP status 기준으로 나뉘었다. 로그인 성공은 `200`, 인증 실패는 `401`로 읽히기 때문에 서버 쪽 의도가 더 분명해졌다.
  - 대신 프론트에 `fetch` 코드가 들어가면서 단순 정적 HTML보다 한 단계 복잡해졌다. 앞으로는 HTML form 기본 동작과 JS 요청 제어 중 어떤 방식을 택할지 먼저 정하고 구현해야 한다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  - 로그인 버튼을 눌러도 요청이 가지 않는 것처럼 보였다. 서버 라우트나 로그인 로직을 의심했지만, 실제 문제는 프론트 쪽에 있었다.
- 처음 추측한 원인:
  - `/user/login` 라우팅이 잘못됐거나, `UserController.login()`에서 응답이 제대로 만들어지지 않는다고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - `login/index.html`의 form, button, script를 다시 봤다. 서버 코드보다 먼저 브라우저에서 submit 이벤트가 발생하는지 확인해야 했다.
  - JS는 form의 `submit` 이벤트에 fetch를 연결해두었는데, 버튼의 타입이 무엇인지 확인했다.
- 실제 원인:
  - 로그인 버튼이 `type="button"`이었다. 이 타입은 클릭해도 form submit을 발생시키지 않는다. JS는 `submit` 이벤트만 기다리고 있었기 때문에 fetch 요청도 실행되지 않았다.
  - form 태그에도 `class="form", method="post", action="/user/login"`처럼 콤마가 들어가 있었다. 동작에 직접적인 원인은 아니더라도 HTML 구조상 정리해야 할 부분이었다.
- 해결 방법:
  - 버튼 타입을 `submit`으로 바꿨다.
  - form 태그의 콤마를 제거하고 `class`, `method`, `action` 속성을 정상적인 HTML 형태로 정리했다.
  - 로그인 실패 시 보여줄 문구 영역을 추가하고, `401`을 받으면 문구를 보이게 하면서 비밀번호 입력값을 비우도록 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 서버 라우트부터 의심하기 전에 브라우저 이벤트가 실제로 발생하는지 먼저 본다.
  - form submit을 JS로 가로챌 때는 button type, form action, method, event listener가 서로 맞는지 확인한다.
  - HTML 속성 오타나 잘못된 태그 구조도 요청이 안 가는 원인이 될 수 있으니 같이 본다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
  - 기능을 바로 붙이기보다, step5를 시작하기 전에 라우터와 패키지 구조를 먼저 정리했다. `RequestHandler`에 계속 분기를 추가하지 않고, `RequestRouter`와 `UserController` 쪽으로 책임을 나눴다.
  - `StaticResourceHandler`에 몰려 있던 역할도 한 번에 다 들고 가지 않고, 파일 해석과 파일 읽기, 응답 조립으로 나눴다.
  - 로그인 구현 중에도 `302`와 `200` 중 어떤 응답이 지금 화면 흐름에 맞는지 다시 따져봤다. 단순히 “로그인 성공은 redirect”라고 밀어붙이지 않고, 실패 메시지를 같은 화면에 보여주려는 목표까지 같이 봤다.
- 그 접근이 도움이 된 이유:
  - 구조를 먼저 정리해두니 로그인 라우트를 추가할 때 손댈 곳이 비교적 명확했다. `Routes`에 경로를 추가하고, `RequestRouter`에서 `UserController`로 연결하면 됐다.
  - `HttpResponse`도 처음에는 `location`, `setCookie` 필드를 따로 들고 있었지만, 헤더가 늘어날수록 필드가 계속 늘어나는 문제가 보였다. 그래서 `headers Map`으로 바꾸는 쪽이 더 유지보수에 맞다고 판단할 수 있었다.
  - 응답 상태 코드를 화면 동작과 함께 생각하면서, HTTP를 단순히 서버 응답 문자열이 아니라 브라우저와 프론트 코드가 해석하는 약속으로 보게 됐다.
- 다음에도 반복하고 싶은 점:
  - 새 기능을 붙이기 전에 지금 구조에서 분기가 어디로 늘어나는지 먼저 본다.
  - 이름이 역할을 설명하지 못하면 바로 고친다. `HandlerMapping`, `queryParams`, `in`, `br`처럼 처음에는 괜찮아 보여도 단계가 넘어가면 어긋나는 이름이 생긴다.
  - HTTP 상태 코드나 헤더처럼 의미가 있는 값은 화면 동작까지 연결해서 판단한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 로그인 버튼이 안 눌리는 문제를 처음부터 서버 쪽으로 생각했다. 실제 원인은 HTML button type이었는데, 브라우저 이벤트 흐름을 먼저 확인하지 않았다.
  - `Set-Cookie` 값을 만들 때 헤더 이름과 값의 형태를 더 꼼꼼히 봐야 했다. 쿠키는 body가 아니라 응답 header에 들어가야 하고, `sid=세션ID; Path=/` 형태가 정확해야 한다.
  - `HttpResponse`를 `headers Map`으로 바꾸면서도 어느 메서드까지 headers를 받게 할지 조금 흔들렸다. `redirect`에 headers를 받을지, `Location`은 메서드 인자로 둘지 같은 기준을 늦게 정했다.
- 그렇게 된 원인:
  - 서버 구현에 집중하다 보니 HTML form과 JS 이벤트 흐름을 상대적으로 가볍게 봤다.
  - 쿠키와 세션은 서버에만 있는 기능처럼 생각했지만, 실제로는 브라우저가 `Set-Cookie`를 받아 저장하고 다음 요청에 `Cookie`를 붙이는 흐름까지 같이 봐야 했다.
  - 응답 객체를 일반화할 때도 “지금 필요한 것”과 “나중에 확장될 것” 사이의 선을 바로 긋지 못했다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 브라우저에서 동작하지 않는 문제는 먼저 HTML 이벤트, form submit, network 요청 순서로 확인한다.
  - 쿠키 관련 기능을 구현할 때는 응답 헤더와 다음 요청 헤더를 한 쌍으로 본다. `Set-Cookie`를 보냈는지, 이후 `Cookie`가 들어오는지까지 확인한다.
  - 리팩토링할 때는 public method를 무작정 일반화하지 않는다. `redirect`처럼 필수 의미가 있는 값은 메서드 인자로 남기고, 부가 정보만 headers로 받을지 판단한다.
