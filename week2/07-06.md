# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - GET 회원가입에서 시작해 POST 회원가입까지 흐름을 확장했다. 처음에는 query string에서 값을 꺼내 `User`로 저장했고, 이후 POST 요청에서는 body를 `Content-Length`만큼 읽어 `application/x-www-form-urlencoded` 값을 파싱하도록 바꿨다.
  - URL encoding된 한글, 이메일 값을 `URLDecoder`로 복원했다. `김asd현eqwe`처럼 한글과 영문이 섞인 값도 문자열 전체를 한 번 디코딩하면 된다는 점을 확인했다.
  - `/user/create` 경로를 회원가입 기능 경로로 보고, GET 요청은 막고 POST 요청만 처리하도록 정리했다.
  - 회원가입 성공 후 `302 Found`와 `Location: /index.html`을 내려 브라우저 주소가 `/index.html`로 바뀌도록 redirect를 구현했다.
  - `HttpMethod`, `ContentType`, `HandlerMapping`, `HttpResponse`를 정리하면서 `RequestHandler`에 있던 책임을 조금씩 덜어냈다.
  - 리뷰 피드백을 반영해 디렉터리 요청의 Content-Type 판별 기준을 실제 응답 파일 기준으로 바꿨고, 인메모리 DB는 `ConcurrentHashMap`으로 변경했다.
  - request parameter라는 이름이 GET query와 POST body를 함께 설명하기에 더 맞다고 판단해 `queryParams`를 `requestParams`로 정리했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - step5에서 로그인 기능이 들어오면 `HandlerMapping`의 if 분기가 계속 늘어날 수 있다. 지금은 문서로만 남겼고, 실제 route map 구조는 아직 적용하지 않았다.
  - POST body 파싱은 현재 `Content-Length`와 `application/x-www-form-urlencoded` 흐름에 맞춰 구현했다. multipart, JSON, body read 실패 같은 경우는 아직 다루지 않았다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - `RequestHandler`는 socket 입출력과 요청/응답 연결만 맡기고, method와 path를 보고 어떤 처리를 할지는 `HandlerMapping`으로 옮기기로 했다.
- 그렇게 판단한 이유:
  - step3에서는 `/user/create` 하나만 처리하면 됐지만, step4에서 POST와 redirect가 들어오면서 `RequestHandler` 안의 분기가 빠르게 복잡해졌다. 앞으로 로그인, 로그아웃, 마이페이지가 들어오면 같은 문제가 더 커질 것 같았다.
  - HTTP method를 enum으로 알고 있다는 것과 그 요청을 실제로 처리할 수 있다는 것은 다르다. 그래서 `GET`, `POST` 자체보다 `method + path` 조합을 기준으로 봐야 한다고 판단했다.
- 고려했지만 선택하지 않은 대안:
  - `UNKNOWN` method만 막고 나머지는 통과시키는 방식도 생각했다. 하지만 그렇게 하면 `DELETE /index.html` 같은 요청이 정적 파일 처리로 흘러갈 수 있다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - `RequestHandler`는 더 얇아졌고, 요청을 어디서 처리하는지 볼 때 `HandlerMapping`부터 확인하면 되는 구조가 됐다.
  - 테스트도 GET query 기반에서 POST body 기반으로 바뀌면서, 실제 브라우저 form 전송 흐름과 더 가까워졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - GET 회원가입을 POST로 바꾼 뒤에도 처음에는 기존 query parameter 방식으로 생각했다. 그런데 POST form 요청은 값이 URL 뒤가 아니라 request body에 들어간다는 점 때문에 저장 흐름이 맞지 않았다.
- 처음 추측한 원인:
  - `/user/create` 경로 매핑이나 form action이 잘못된 줄 알았다.
- 원인을 확인하기 위해 시도한 방법:
  - 요구사항의 HTTP Request 예시를 다시 봤고, header 뒤 빈 줄 다음부터 body가 시작된다는 점을 확인했다.
  - `Content-Length`가 body 길이를 알려준다는 점을 기준으로 파서가 어디까지 읽고 있는지 확인했다.
- 실제 원인:
  - GET은 request line의 query string을 파싱하면 되지만, POST는 header를 모두 읽은 뒤 body를 별도로 읽어야 했다. 기존 구조는 이 차이를 충분히 반영하지 못했다.
- 해결 방법:
  - `Content-Length` header를 파싱하고, POST 요청일 때 body를 그 길이만큼 읽도록 했다.
  - body도 `userId=...&password=...` 형태라 기존 parameter 파싱 로직을 재사용했다.
  - 회원가입 성공 응답은 HTML body 대신 `302 Found`와 `Location: /index.html`을 내려 redirect로 처리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 요청 method만 보지 말고, 값이 request line, header, body 중 어디에 실리는지 먼저 확인한다.
  - 브라우저가 보내는 실제 HTTP 요청 예시를 보고 구현 기준을 잡는다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 회원가입 후 redirect 상태 코드를 정할 때 AI의 제안을 그대로 따르지 않고, 요구사항과 관련 문서를 다시 확인했다. 처음에는 평소 익숙한 대로 `302 Found`를 썼다가, AI가 POST 이후 이동에는 `303 See Other`가 더 적절하다고 해서 잠깐 흔들렸다. 그래도 요구사항에서 `302`를 명시하고 있었고, 실제 웹에서도 POST 이후 redirect에 `302`를 쓰는 경우가 많다는 점을 확인한 뒤 최종 구현은 다시 `302 Found`로 맞췄다.
- 그 접근이 도움이 된 이유:
  - AI의 설명이 그럴듯해도 요구사항과 다르면 결국 맞는 구현이 아니다. 이번에는 제안을 그대로 받아들이지 않고 한 번 더 확인해서, 과제에서 요구하는 동작과 실제 구현을 맞출 수 있었다.
  - 표준이나 관례가 얽힌 내용은 답이 하나처럼 보이기 쉽다. 근거를 확인해 두니 왜 `302`를 선택했는지 스스로 납득할 수 있었고, 리뷰를 받을 때도 그 판단을 설명할 수 있게 됐다.
- 다음에도 반복하고 싶은 점:
  - AI는 구현을 도와주는 도구로 쓰되, 최종 판단은 요구사항과 근거를 확인한 뒤 직접 내린다. 특히 HTTP status code처럼 표준과 관례가 함께 걸린 내용은 먼저 문서를 확인하고 결정한다.
  - 그리고 기본적으로 알아야 할 지식들이 많은데 부족한 상황이라는 걸 알게 된 거 같아서, 이런 느낌이 들 때마다 기록해서 공부하려고 한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 처음에는 GET query parameter와 POST body parameter를 같은 방식으로 생각했다. 그래서 `queryParams`라는 이름도 오래 끌고 갔다.
  - redirect도 처음에는 `/`로 보내도 된다고 생각했지만, 요구사항은 브라우저 URL이 `/index.html`로 바뀌는 것이었다.
  - `HandlerMapping`을 if문으로 계속 확장할 때 생길 문제를 알면서도, 어느 시점에 route map 구조로 바꿀지 기준을 바로 세우지는 못했다.
- 그렇게 된 원인:
  - 동작이 되는지에 먼저 집중했고, HTTP 요청의 의미나 요구사항 문장을 충분히 엄밀하게 보지 않았다.
  - step3에서 만든 구조를 step4에 그대로 이어 쓰려다 보니 이름과 책임이 조금씩 어긋났다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 구현 전에 요구사항의 HTTP 예시를 실제 파싱 단위로 나눠 본다. request line, header, 빈 줄, body를 먼저 구분한다.
  - 변수명은 기능이 바뀌는 순간 같이 다시 본다. GET query만 다룰 때의 이름이 POST body까지 설명할 수 있는지 확인한다.
  - 분기가 세 개 이상 늘어날 것 같으면 바로 구조 개선 후보로 적어두고, 이번 단계에서 할지 다음 단계로 미룰지 결정한다.
