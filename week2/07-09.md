### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:
  - Step6 요구사항에 맞춰 세션 정보를 기준으로 HTML 헤더를 동적으로 렌더링했다. 비로그인 상태에서는 로그인/회원가입 버튼을 보여주고, 로그인 상태에서는 사용자 이름, 글쓰기, 로그아웃 버튼을 보여주도록 만들었다.

  - `PageRoutes`를 추가해 `/`, `/index.html`, `/login`, `/registration`, `/mypage` 같은 페이지 경로를 한 곳에서 관리하게 했다.
  - `AuthGuard`를 추가해 로그인 상태에 따른 페이지 접근 제어를 구현했다. 비로그인 사용자가 `/mypage`에 접근하면 `/login`으로 보내고, 로그인 사용자가 `/login`, `/registration`에 접근하면 `/index.html`로 보내도록 했다.
  - 헤더를 사용하는 HTML 페이지에 `{{AUTH_HEADER}}` placeholder를 적용했고, HTML 페이지는 `DynamicPageHandler`, `HtmlTemplateRenderer`를 거쳐 응답하도록 일반화했다.
  - `/mypage` 닉네임 입력 칸에 로그인한 사용자의 이름이 들어가도록 `MyPageRenderer`를 추가했다.
  - Step6 동작을 검증하는 테스트를 추가했다. 비로그인/로그인 상태의 index 헤더, 공통 HTML 헤더 치환, mypage 접근 제어, 로그인 상태의 로그인/회원가입 페이지 접근 차단, mypage 닉네임 표시를 확인했다.
  - 리뷰에서 지적된 사용자 이름 렌더링의 XSS 위험을 보완했다. `HtmlEscaper`를 추가해 헤더와 mypage input value에 들어가는 사용자 이름을 HTML escape 처리했다.
  - `/mypage?x=1`처럼 query string이 붙은 보호 페이지 요청도 접근 제어를 우회하지 못하도록 `PageRoutes.matches()`에서 query string을 제거한 뒤 비교하게 했다.
  - PR 본문 초안을 작성했고, Step6에서 어떤 작업을 했는지와 어떤 고민이 있었는지 정리했다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  - HTML 전체를 Java 코드로 새로 만들지 않고, 정적 HTML에 placeholder를 두고 서버에서 필요한 부분만 치환하는 방식으로 구현했다. 그중 공통 헤더는 `HtmlTemplateRenderer`에서 처리하고, mypage 전용 값은 `MyPageRenderer`로 분리했다.
- 그렇게 판단한 이유:
  - Step6에서 로그인 전후로 달라지는 핵심은 페이지 전체가 아니라 헤더 영역이었다. `/index.html`과 `/main/index.html`을 별도 페이지처럼 나눠 쓰면 같은 구조의 HTML을 중복해서 관리하게 될 가능성이 컸다.
  - 처음에는 index 전용 렌더러만 둬도 된다고 생각했지만, 로그인/회원가입/mypage/article/comment 페이지도 같은 헤더를 사용하고 있었다. 한 페이지에서만 동적 헤더를 처리하면 화면마다 로그인 상태 표시가 달라질 수 있었다.
  - mypage는 지금은 닉네임만 치환하지만, 이후 프로필 이미지가 들어오면 전용 placeholder가 더 늘어날 가능성이 높았다. 그래서 공통 렌더러에 모든 값을 넣기보다 `MyPageRenderer`를 따로 두는 쪽을 선택했다.
- 고려했지만 선택하지 않은 대안:
  - 로그인 전용 HTML과 로그인 후 전용 HTML을 따로 두는 방법도 있었다. 구현은 단순해 보이지만, 같은 헤더와 레이아웃을 여러 파일에서 관리해야 해서 수정 누락이 생기기 쉽다고 봤다.
  - `HtmlTemplateRenderer` 하나에서 `{{AUTH_HEADER}}`, `{{USER_NAME}}`, 이후 프로필 이미지까지 모두 처리하는 방법도 가능했다. 하지만 페이지 전용 값이 늘어나면 공통 렌더러가 금방 지저분해질 것 같았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - HTML 페이지라면 공통 흐름으로 헤더를 치환하게 되어, 특정 페이지에서만 로그인 상태 표시가 빠지는 문제를 줄일 수 있었다.
  - mypage 전용 값은 `MyPageRenderer`에 모이게 됐다. 이후 프로필 이미지 placeholder가 생겨도 수정 위치가 비교적 분명하다.
  - 테스트에서도 index만 확인하지 않고, 다른 HTML 페이지의 placeholder 치환과 mypage 닉네임 표시까지 같이 검증할 수 있었다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  - 세션 쿠키 문자열을 직접 조립하던 코드를 `CookieBuilder`와 `SessionCookie`로 분리하는 과정이 가장 헷갈렸다. 빌더 패턴을 직접 적용해본 경험이 거의 없어서, 어디까지를 빌더가 책임지고 어디부터를 세션 쿠키 도메인 규칙으로 봐야 하는지 바로 잡히지 않았다.
  - 처음에는 `AuthController`에서 `Set-Cookie` 값을 만들고 있었는데, 로그인 성공 쿠키와 로그아웃 만료 쿠키가 같은 형식을 공유하면서도 세부 값은 달랐다. 이걸 단순 문자열 메서드로 뺄지, 쿠키 전용 객체를 둘지 판단하는 데 시간이 걸렸다.
- 처음 추측한 원인:
  - 처음에는 쿠키 문자열 조립이 어렵다기보다, 문법을 어디에 둘지 정하지 못해서 헷갈린다고 생각했다. `sid=...; Path=/`, `sid=; Max-Age=0; Path=/`처럼 결과 문자열은 단순했기 때문이다.
  - 그런데 막상 코드를 나누려고 보니 문제는 문자열 자체가 아니라 책임의 경계였다. `CookieBuilder`가 `sid`라는 세션 쿠키 이름까지 알아도 되는지, `Max-Age=0` 같은 로그아웃 규칙을 빌더가 알아도 되는지 애매했다.
- 원인을 확인하기 위해 시도한 방법:
  - 로그인 성공과 로그아웃 응답에서 실제로 필요한 `Set-Cookie` 값을 먼저 비교했다. 공통으로 필요한 것은 `name=value`와 `Path=/` 같은 쿠키 조립 규칙이었고, 세션 쿠키 이름이나 만료 규칙은 별도 정책에 가까웠다.
  - 그래서 `CookieBuilder`는 `of(name, value)`, `path(path)`, `maxAge(age)`, `build()`처럼 쿠키 문자열을 조립하는 일만 맡기고, `SessionCookie`가 `sid` 이름과 로그인/로그아웃 쿠키 생성 규칙을 갖도록 나눴다.
  - `AuthController`에서는 더 이상 쿠키 문자열을 직접 만들지 않고, `SessionCookie.create(session)`과 `SessionCookie.expire()`를 호출하게 했다.
- 실제 원인:
  - 빌더 패턴이 익숙하지 않다 보니, 처음에는 빌더를 단순히 문자열을 예쁘게 만드는 도구처럼만 봤다. 하지만 이번 코드에서는 메서드 체이닝보다 책임 분리가 더 중요했다.
  - `CookieBuilder`가 세션 정책까지 알면 이름만 빌더일 뿐, 결국 인증 흐름에 강하게 묶인 클래스가 된다. 반대로 `SessionCookie`가 문자열 조립 방식까지 다 알면 쿠키 속성이 늘어날 때마다 같은 문자열 조립 코드가 반복될 수 있다.
- 해결 방법:
  - `CookieBuilder`는 쿠키 형식 조립만 담당하게 했다. `name=value`로 시작하고, 필요한 속성은 `path`, `maxAge` 같은 메서드로 붙인다.
  - `SessionCookie`는 세션 쿠키 정책을 담당하게 했다. 쿠키 이름은 `sid`로 고정하고, 로그인 성공 시에는 세션 아이디와 `Path=/`를 담은 쿠키를 만들고, 로그아웃 시에는 빈 값과 `Max-Age=0`, `Path=/`를 담은 만료 쿠키를 만들게 했다.
  - 이 구조로 바꾸고 나니 `AuthController`는 로그인 결과를 보고 응답 헤더에 `Set-Cookie`를 넣는 흐름만 남게 됐다. 쿠키 문자열의 세부 형식은 컨트롤러 밖으로 빠졌다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 빌더를 만들기 전에 먼저 변하는 값과 고정되는 규칙을 구분한다.
  - 빌더는 형식 조립을 맡기고, 도메인 정책은 별도 클래스로 빼는 편이 나은지 확인한다.
  - 메서드 체이닝이 가능하다는 이유만으로 빌더를 만들지 말고, 호출하는 쪽의 책임이 실제로 줄어드는지 본다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
  - 처음에 `UserController`와 `AuthController` 쪽에 몰려 있던 책임을 그냥 두지 않고, `UserService`, `AuthService`, `SessionCookie`, `CookieBuilder`로 단계적으로 분리했다.
  - 회원가입은 사용자 생성 흐름으로 보고 `UserService`에 맡겼고, 로그인 검증과 세션 생성/조회는 `AuthService`로 옮겼다. 컨트롤러는 요청 파라미터를 확인하고, 서비스 결과를 HTTP 응답으로 바꾸는 역할에 더 가깝게 정리했다.
- 그 접근이 도움이 된 이유:
  - 처음 구조에서는 컨트롤러가 요청 파라미터 확인, 사용자 생성, 비밀번호 검증, 세션 생성, 쿠키 문자열 조립까지 많이 알고 있었다. 기능은 동작해도, 다음 요구사항이 붙으면 컨트롤러가 계속 커질 수밖에 없는 구조였다.
  - 책임을 나누고 나니 각 변경의 위치가 조금 더 분명해졌다. 사용자 생성 규칙은 `UserService`, 로그인 성공/실패 판단은 `AuthService`, 세션 쿠키 생성 규칙은 `SessionCookie`에서 보면 된다.
  - 이 분리가 Step6의 동적 HTML 작업에도 도움이 됐다. `RequestRouter`에서 sid를 꺼낸 뒤 `AuthService.findLoginUser(sessionId)`로 로그인 사용자를 조회할 수 있어서, 렌더링 쪽에서는 로그인 사용자 객체만 보고 판단하면 됐다.
- 다음에도 반복하고 싶은 점:
  - 컨트롤러에 로직이 쌓이기 시작하면, 기능이 더 커지기 전에 서비스나 전용 객체로 옮길 수 있는지 먼저 본다.
  - 단순히 파일을 나누는 데서 끝내지 않고, 나눈 뒤에 호출하는 쪽의 책임이 실제로 줄었는지 확인한다.
  - 문자열 조립, 세션 정책, 사용자 생성, 로그인 검증처럼 성격이 다른 일을 한 클래스에 오래 두지 않는다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 동적 HTML을 구현하면서 사용자 입력값이 HTML에 들어가는 문맥을 처음부터 충분히 고려하지 못했다. 닉네임은 단순 문자열이라고 생각했지만, HTML에서는 문자열이 아니라 문서 구조를 바꿀 수 있는 값이었다.
  - `/mypage?x=1` 같은 query string이 붙은 보호 페이지 요청도 처음 설계 때 테스트하지 않았다. 정상 경로인 `/mypage`만 생각하고 접근 제어를 검증했다.
  - Step6 문서와 PR 문서, 스킬 산출물처럼 커밋할 문서와 개인 작업용 문서가 섞여 untracked로 남았다. 커밋 대상 정리가 조금 늦었다.
- 그렇게 된 원인:
  - 기능 요구사항을 먼저 만족하는 데 집중하면서, 사용자 입력값이 렌더링되는 문맥과 URL 변형 케이스를 뒤늦게 봤다.
  - 정적 HTML을 서버에서 치환하는 방식이 익숙하지 않다 보니, placeholder 치환 자체에 신경이 더 갔다. 치환된 값이 HTML에서 어떤 의미를 갖는지까지 바로 이어서 생각하지 못했다.
  - 문서 작업을 여러 번 나눠 하면서 PR용 문서, 학습용 문서, 스킬 결과물이 한 작업 트리에 같이 남았다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 동적 HTML을 만들 때는 placeholder 목록을 만든 뒤, 각 값이 text인지 attribute인지 URL인지 먼저 구분한다.
  - 인증이 걸린 페이지는 기본 경로뿐 아니라 query string, trailing slash, `index.html` 형태까지 테스트 케이스에 넣는다.
  - 커밋 전에 `git status --short`를 보고, PR에 들어갈 파일과 개인 기록으로 남길 파일을 바로 분리한다.
