### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 책임과 기능으로 잘게 쪼개놨던 코드를 합칠 수 있는 부분에서는 합치기로 결정했다.
- 그렇게 판단한 이유: 객체 지향을 공부하며, 책임과 기능으로 코드를 분리해보며 연습했던 것이었지만, 너무 과도하게 쪼개다보니 나조차도 구조를 파악하기 어렵게 되어버렸다. 그리고 코드를 쪼갰던 이유는 확장성도 있지만, 남들이 봤을 때 쉽게 이해되어야하는데 오히려 더욱 이해하기 어려운 코드가 되었다고 판단하여 다시 합치기로 결정했다.
- 고려했지만 선택하지 않은 대안: 기존의 패키지명 혹은 클래스명, 메서드명을 읽기 쉽고 알아보기 쉽게 바꾸려고 했지만, 그것보다도 구조를 개선하는게 우선이라고 판단하였다. 일단은 구조에 대해서 다시 고민해보고 메서드명도 바꿀 것 같다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 아직 모든 구조가 변경된 것이 아니어서 결과는 알 수 없지만, 최종적으로 목표하는 것은 나도, 남도 모두 읽기 좋은 코드로 만드는 것이다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 요청마다 `RequestHandler`, `HttpRequestParser`, `RequestRouter`와 그 내부의 controller, service 등이 새로 생성되는 구조를 보며, 매번 생성되는 객체를 모두 `static`으로 바꿔야 하는지 고민했다.
- 처음 추측한 원인: 요청마다 같은 객체를 생성하므로 서버에 부하가 생기고 성능이 떨어질 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 각 객체가 요청마다 달라지는 상태를 가지고 있는지, 여러 요청에서 함께 사용해도 되는지 기준으로 역할과 필드를 다시 확인했다. 또한 실제 병목이 객체 생성인지 socket I/O, 파일 읽기, DB 접근 같은 작업인지도 함께 비교했다.
- 실제 원인: 문제는 단순히 객체가 반복 생성된다는 사실이 아니라, 애플리케이션 전체에서 공유해도 되는 객체와 요청마다 새로 만들어야 하는 객체를 구분하지 않은 구조였다. 모든 객체를 `static`으로 바꾸면 요청별 상태까지 공유될 수 있어 thread-safe 문제가 생길 수 있었다.
- 해결 방법: `RequestRouter`와 mutable field가 없는 `HttpRequestParser`는 `WebServer`에서 한 번 생성해 `RequestHandler`에 생성자로 주입했다. 반면 `HttpRequest`, `HttpResponse`, `Session`처럼 요청 또는 결과를 표현하는 객체는 요청마다 새로 만들도록 유지했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 객체 생성 횟수만 보고 `static` 전환을 결정하지 않고, 해당 객체가 상태를 가지는지, 공유해도 안전한지, 실제 병목인지부터 확인한다. 특히 공유 객체에는 요청별 상태를 필드로 추가하지 않는지 함께 점검한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 코딩을 하기에 앞서 계획 문서를 AI와 함께 작성하였다.
- 그 접근이 도움이 된 이유: 기존에 잦은 바이브 코딩을 하며 선택했던 방법은 확실한 설계 문서를 작성하고 개발을 맡기는 것이었는데, AI를 개발에 사용하지 않으려고 노력하다보니 계획하고 설계하는 과정조차도 하지 않게 되면서 오히려 완성도가 낮은 코드를 짜게 된 것 같다. 때문에, AI와 함께 계획 문서부터 차근차근 작성하고 개발에는 나의 역량을 최대한 키우려고 하다보니 전보다 속도는 느려도 완성도는 높아진 것 같다.
- 다음에도 반복하고 싶은 점: AI에게도 그렇겠지만, 사람에게도 항상 설계와 계획은 필수적인 부분인 것 같다고 느꼈기 때문에, 앞으로도 개발할 때 설계하고 계획한 뒤 개발하는 방법을 고수할 것 같다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 수업 일정으로 인해 개발을 크게 진행하지 않아서 크게 아쉬웠던 행동은 없었던 것 같다.

## 주간 프롬프트 회고

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:

'''text
step7 요구사항 및 figma에 적힌 요구사항 공유 후

이거 다 확인했을 때
  1. 내가 현재 구조에서 미리 리팩토링 해야할 것들.
  2. 기능 구현해야 할 것들.
  3. PR과 커밋을 어떻게 나눠야 하는지.
  4. 현재 구조 아키텍처
  5. 예상 구조 아키텍처
  6. 보안적으로 챙겨야 할 부분들.

  이런 구조로 계획 문서 하나 작성해줘.

  구조를 조금 변형해도 좋지만, 위의 내용은 꼭 포함돼야하고 추가로 포함해야 될
  내용이 있다면 적절한 위치에 넣어준 뒤에 나한테 알려만 줘.
'''

선택한 이유: 저번 주의 나의 프롬프트의 문제는 구체화되지 않은 요구와 결과를 제공받는 데에 있어서 형식을 지정해주지 않은 것이었다고 생각한다. 때문에 이 프롬프트는 어느정도 그 부분을 의식해서 작성했고, 결과도 내가 원했던 대로 나왔던 것 같다.

### 2. 원본 프롬프트 구조 분석
- 문제 상황: Step7 요구사항과 Figma 화면을 구현하기 전에, 현재 구조에서 무엇을 정리하고 어떤 순서로 개발할지 판단할 필요가 있었다. 요구 범위가 넓어 바로 구현하면 구조가 더 복잡해질 수 있다는 우려도 있었다.
- 원하는 결과를 어떻게 요구했는가: 사전 리팩토링, 구현 기능, PR과 커밋 분리, 현재·예상 아키텍처, 보안 항목을 여섯 가지로 나눠 계획 문서에 포함해 달라고 요청했다.
- 필요한 맥락이 포함되어 있는가: Step7 요구사항과 Figma 내용을 먼저 공유한다는 전제가 있어 작업 범위는 전달할 수 있었다. 다만 현재 코드 구조, 이미 구현된 기능, 팀의 개발 방식은 프롬프트 안에 직접 적지 않았다.
- 판단 기준이나 제약 조건이 있는가: 제시한 여섯 항목은 반드시 포함하도록 했고, 구조는 조금 바꿔도 되지만 추가할 내용이 있다면 적절한 위치에 넣어 달라고 했다. 다만 과도한 리팩토링을 피해야 한다는 기준이나 우선순위는 더 분명히 적을 수 있었다.
- 출력 방식이 명확한가: 계획 문서 하나를 작성해 달라고 했고 필수 항목도 제시했다. 그러나 PR별로 어느 정도까지 상세히 작성할지, 표나 목록 중 어떤 형식을 원하는지는 정하지 않았다.
- 나의 가설, 우려, 의도가 드러나는가: Step7을 구현하기 전에 구조와 작업 순서를 점검하고 싶다는 의도는 드러난다. 특히 PR과 커밋을 나눠 달라고 한 부분에서 큰 기능을 한 번에 처리하지 않으려는 생각도 확인할 수 있다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
Step7 요구사항과 Figma 화면 정보를 공유한 뒤, 현재 프로젝트 코드를 기준으로 Step7 구현 계획 문서를 작성해줘.

현재 목표는 기능을 빠르게 추가하면서도, 기존 구조를 더 복잡하게 만들지 않는 것이다. 새로운 클래스나 추상화는 실제로 필요한 경우에만 제안하고, 단순히 구조를 예쁘게 만들기 위한 리팩토링은 제외해줘.

문서에는 아래 내용을 포함해줘.

1. 요구사항 해석과 구현 범위
   - Figma 화면과 현재 정적 파일 경로가 다르면 현재 프로젝트 경로 기준으로 정리
   - 이번 Step에서 구현할 것과 다음 단계로 미룰 것을 구분

2. 현재 구조 평가와 사전 리팩토링
   - 현재 요청 처리 흐름을 기준으로 유지할 부분과 수정할 부분 제안
   - 새 구조를 제안할 때는 도입 이유, 기대 효과, 지금 도입하지 않아도 되는 이유를 함께 작성

3. 기능 구현 계획
   - 게시판, 댓글, 좋아요, 이미지 업로드, mypage, 에러 응답, H2/JDBC, Javadoc을 기능 단위로 정리
   - 기능별 주요 클래스와 수정 범위를 간단히 설명

4. PR과 커밋 계획
   - PR은 독립적으로 검토하고 테스트할 수 있는 단위로 나누기
   - 각 PR의 목적, 포함 범위, 제외 범위, 예상 커밋을 작성

5. 아키텍처와 품질 기준
   - 현재 구조와 Step7 이후 예상 구조를 요청 흐름 중심으로 설명
   - 공유 객체와 요청별 객체의 생명주기, 예외 처리, 보안, 테스트에서 확인할 점을 정리

확정적으로 도입할 항목과 구현 중 필요성을 다시 판단할 항목을 구분해서 작성해줘. 문서 마지막에는 리스크와 보류 항목도 정리해줘.
```

### 4. AI가 개선한 프롬프트 분석
- 원본 프롬프트는 필요한 산출물을 여섯 항목으로 정리해 계획 문서의 방향을 잡아줬다. 개선안은 여기에 현재 프로젝트 코드를 기준으로 판단해 달라는 맥락을 추가했다.
- 가장 큰 차이는 리팩토링의 기준을 명시한 점이다. 기능을 추가하되 구조가 앞서가지 않도록 하고, 새 클래스나 추상화는 필요성이 확인될 때만 제안하도록 제한했다. 처음 계획이 구조 중심으로 커질 수 있었던 문제를 줄일 수 있다.
- 구현 범위와 보류 범위를 구분하도록 요청해 Step7에서 반드시 끝낼 일과 나중에 판단할 일을 분리했다. 계획 문서가 구현 단계에서 과도한 약속이 되는 것을 막는 데 도움이 된다.
- PR과 커밋 계획에는 각 단위의 목적, 포함 범위, 제외 범위, 예상 커밋을 적도록 했다. 단순히 PR을 나누는 것보다 검토와 테스트 가능한 단위로 나누는 기준이 더 분명해진다.
- 아키텍처 설명도 패키지나 클래스 목록 대신 요청 흐름, 객체 생명주기, 예외 처리, 보안, 테스트 관점에서 작성하도록 바꿨다. 실제 구현 중 확인해야 할 기준이 함께 남으므로 계획을 코드로 옮기기 쉬워질 것으로 예상한다.
