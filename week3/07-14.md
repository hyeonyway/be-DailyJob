- 오늘 실제로 끝낸 기능/수정/학습:
  - `Comment` 모델과 `comments` 테이블, JDBC 저장소를 추가했다. 댓글 저장, 게시글별 전체 조회, 먼저 작성된 댓글 3개 조회, 댓글 수 조회까지 구현하고 H2 테스트로 확인했다.
  - 게시글과 댓글 조회를 service로 분리하고, 작성자 정보를 한 번에 가져오는 사용자 일괄 조회를 추가했다. 메인 화면에서는 handler가 조회 결과를 화면 데이터로 조립하고 renderer가 HTML을 만드는 구조로 정리했다.
  - 로그인 사용자의 게시글 작성과 댓글 작성 흐름을 연결했다. 댓글 작성은 GET `/comment?articleId={id}`로 화면을 열고 같은 URL에 POST한 뒤, 원래 게시글로 돌아가도록 구현했다.
  - 메인 화면에서 게시글과 댓글 3개를 동적으로 표시하고, `showAllComments=true`일 때 해당 게시글의 댓글을 모두 보여주도록 구현했다.
  - 저장소, service, renderer, 요청 흐름을 나눠 테스트했고 전체 테스트 통과를 확인했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 좋아요 증가와 이전 글·다음 글 이동은 아직 메인 화면 동작에 연결하지 못했다.
  - 이미지 업로드와 multipart 요청 처리는 다음 PR 범위로 남아 있다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - repository에서 `ResultSet`을 service로 넘기지 않고 `Article`, `Comment`, `User` 같은 도메인 객체를 반환하기로 했다. 메인 화면에 필요한 데이터는 `MainPageHandler`가 service를 호출해 조립하고, renderer에는 화면 데이터만 전달했다.
- 그렇게 판단한 이유:
  - `ResultSet`이 service까지 올라오면 service가 테이블의 컬럼명과 JDBC 사용법을 알아야 한다. 저장소 구현이 화면 흐름까지 새어 나오는 구조라 결합도가 높아진다. repository 안에서 매핑을 끝내면 service는 게시글과 댓글을 어떻게 사용할지만 다룰 수 있다.
- 고려했지만 선택하지 않은 대안:
  - renderer나 handler가 repository를 직접 호출하는 방법, 작성자마다 사용자를 한 명씩 조회하는 방법도 생각했다. 전자는 화면 코드가 SQL 저장 구조에 묶이고, 후자는 댓글이 늘어날수록 조회 횟수가 함께 늘어나므로 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - JDBC 매핑은 repository 테스트에서, 조회 흐름은 service 테스트에서, HTML 변환은 renderer 테스트에서 따로 확인할 수 있게 됐다. 게시글과 댓글 작성자를 `findByUserIds()`로 한 번에 조회해 메인 화면의 N+1 조회도 피했다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - POST `/comment`를 라우터에 등록한 뒤 GET `/comment?articleId=...`로 댓글 작성 화면에 접근하면 `405 Method Not Allowed`가 반환됐다. 폼을 제출했을 때는 댓글 내용이 없다고 판단해 `400`이 반환되기도 했다.
- 처음 추측한 원인:
  - GET과 POST가 같은 경로를 사용하는 방식 자체가 현재 라우터에서 불가능한 것으로 생각해 `/comment/create` 같은 별도 경로가 필요한지 고민했다.
- 원인을 확인하기 위해 시도한 방법:
  - `RequestRouter`가 route와 정적 파일을 판별하는 순서를 따라가고, 브라우저가 보낸 요청 line과 form body를 로그로 확인했다. 댓글 HTML의 form과 textarea 속성도 요청 parameter 이름과 대조했다.
- 실제 원인:
  - 라우터가 등록된 POST 경로라는 이유로 GET 요청을 먼저 405로 처리하고 있었다. 댓글 내용이 누락된 원인은 textarea에 `name="content"`가 없어 form body에 값이 포함되지 않은 것이었다.
- 해결 방법:
  - 라우터가 메서드와 경로가 정확히 일치하는 동적 route를 먼저 찾고, 그다음 정적 페이지 존재 여부를 확인하도록 순서를 조정했다. GET은 댓글 작성 HTML을, POST는 `CommentController`를 호출하도록 나눴다. form은 현재 URL의 query string을 유지한 채 POST하고 textarea에는 `name="content"`를 지정했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 같은 path를 여러 HTTP method가 공유할 때는 route 존재 여부만 보지 않고 `(method, path)` 조합과 fallback 순서를 먼저 확인한다. form 요청이 비어 있으면 서버 파서보다 먼저 `method`, `action`, 입력 요소의 `name`을 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 댓글 작성 변경량이 커졌을 때 그대로 한 커밋에 넣지 않고, 저장과 요청 검증, 화면과 form 연결, 라우팅과 페이지 handler, 통합 테스트로 다시 나눴다. 체크리스트를 갱신하면서 빠져 있던 비로그인 GET 댓글 화면 테스트도 찾아 기존 테스트 커밋에 보강했다.
- 그 접근이 도움이 된 이유:
  - 각 커밋의 목적과 영향 범위가 분명해져 변경 흐름을 따라가기 쉬워졌다. 구현 여부와 테스트 여부를 따로 확인하면서 `AuthGuard`가 동작한다는 추측에 머물지 않고 실제 redirect 응답을 검증할 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 기능이 동작한 뒤에도 커밋하기 전에 변경사항을 책임별로 다시 나눈다. 체크리스트와 테스트 목록을 대조해 정상 흐름뿐 아니라 인증, 잘못된 parameter, 존재하지 않는 데이터 같은 우회 요청도 확인한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 댓글 작성 화면과 POST 요청에서 `articleId`를 각각 검증하면서 같은 숫자 변환 코드가 `CommentPageHandler`와 `CommentController`에 중복됐다. 처음 작성한 테스트에는 비로그인 사용자의 GET 댓글 화면 접근도 빠져 있었다.
- 그렇게 된 원인:
  - 화면 접근과 저장 요청의 책임을 분리하는 데 집중하면서 두 곳에서 반복되는 입력 규칙까지 정리하지 못했다. 기능이 이미 `AuthGuard`로 막힌다는 사실만 보고 해당 경로의 통합 테스트를 별도로 작성하지 않았다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 동일한 입력 규칙이 두 군데 이상 생기면 커밋 전에 공통 값 객체나 작은 parser가 필요한지 검토한다. 인증이 적용된 새 경로를 추가할 때는 로그인·비로그인 GET과 직접 POST 요청을 테스트 목록에 함께 적는다.
