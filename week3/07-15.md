# be-DailyJob

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 잘못된 게시글 ID와 존재하지 않는 게시글을 구분해 각각 400과 404 오류 페이지를 반환하도록 수정했다.
  - 게시글의 이전 글과 다음 글을 조회하고, 이동할 글이 없을 때는 버튼이 비활성화되도록 구현했다.
  - 요청 본문 크기를 제한해 1MiB를 넘으면 413으로 응답하도록 수정했다. 요청 파싱 예외가 가진 상태 코드에 맞춰 오류 페이지를 반환하도록 흐름도 정리했다.
  - 세션 ID가 로그에 노출되지 않도록 Cookie 값을 가리고 요청 파싱, 라우팅, 예외 응답 로그 형식을 맞췄다.
  - RFC 9112를 확인해 Request Start Line의 토큰 수와 method token, request target, HTTP version을 검증했다. 헤더 이름은 대소문자를 구분하지 않고, 콜론 뒤 OWS는 허용하되 헤더 이름 앞이나 콜론 앞 공백은 400으로 거부하도록 수정했다.
  - `Content-Length`만큼 body를 byte 단위로 끝까지 읽고 원본 `byte[]`를 `HttpRequest`에 보관하도록 변경했다. form-urlencoded일 때만 parameter로 해석하도록 읽기와 파싱 책임을 나눴다.
  - `ContentType.matches()`를 추가해 media type의 대소문자와 `charset`, `boundary` 같은 parameter를 분리해서 비교하도록 했다.
  - 요청 원본 body, binary body, 방어적 복사, Content-Type 비교와 기존 예외 응답을 검증하는 테스트를 추가했다.
  - Step7-1에서 구현한 기본 게시판 기능과 현재 구조를 문서로 정리했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - `multipart/form-data`를 part 단위로 파싱하고 업로드 파일을 저장하는 기능은 아직 구현하지 않았다.
  - `HttpHeaders`, `RequestParameters`, `RequestBody` 같은 값 객체를 활용한 구조 리팩토링은 기능 구현이 끝난 뒤 진행할 PR로 계획만 세웠다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - body를 읽는 기준은 HTTP method가 아니라 `Content-Length`로 두고, 읽은 body를 해석하는 기준은 `Content-Type`으로 정했다. `readBody()`는 byte 읽기만 맡기고, form-urlencoded 파싱은 이미 읽은 `byte[]`를 받아 처리하도록 분리했다.
- 그렇게 판단한 이유:
  - POST만 body를 가질 수 있는 것은 아니며 앞으로 PUT이나 PATCH가 추가될 수 있다. parser가 POST에 묶이면 method가 늘 때마다 파서를 수정해야 한다. 이미지 파일은 문자열로 바꾸면 원본이 손상될 수 있으므로 multipart 구현에 앞서 raw body도 보존해야 했다.
- 고려했지만 선택하지 않은 대안:
  - POST 요청에서만 body를 읽고 바로 UTF-8 문자열로 변환해 parameter Map만 남기는 방식도 생각했다. 지금의 회원가입과 글쓰기에는 충분하지만 binary 파일을 처리할 수 없고 body를 두 번 읽을 가능성도 있어 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - HTTP message framing과 body format 해석의 책임이 나뉘었다. 기존 form 요청을 유지하면서 multipart parser가 같은 raw body를 사용할 수 있게 됐고, method가 추가돼도 body 읽기 코드를 손댈 일이 줄었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - `HttpRequest`는 파싱이 끝난 요청을 표현하므로 record가 어울려 보였다. 하지만 내부에 `Map`과 `byte[]`를 넣어도 실제로 불변 객체가 되는지 확신하기 어려웠다.
- 처음 추측한 원인:
  - record의 component가 final이므로 내부 상태도 자동으로 보호될 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - record가 보장하는 불변성의 범위를 확인하고 `Map`과 배열의 변경 가능성, 배열의 `equals()`와 `hashCode()` 동작을 현재 `HttpRequest` 구조와 비교했다.
- 실제 원인:
  - record는 component 참조의 재할당만 막는 얕은 불변성을 제공한다. mutable Map은 여전히 수정할 수 있고, `byte[]` accessor는 원본 배열을 노출한다. 자동 동등성 비교도 배열 내용이 아닌 참조를 기준으로 한다.
- 해결 방법:
  - 이번 변경에서는 일반 class를 유지하고 body를 생성자와 getter에서 각각 복사했다. 이후 별도 PR에서 `HttpHeaders`, `RequestParameters`, `RequestBody`가 규칙과 불변성을 맡도록 리팩토링한 뒤 `HttpRequest`의 record 전환을 다시 검토하기로 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 문법이 간결해지는지만 보지 않고 객체가 가진 값의 변경 가능성, 외부 노출 방식, 동등성 기준부터 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 헤더 공백과 대소문자 처리에 관한 피드백을 바로 코드에 옮기지 않았다. RFC 9112의 field line 규칙을 확인한 다음 테스트로 고정했다.
- 그 접근이 도움이 된 이유:
  - 단순히 `trim()`을 추가했다면 `Host : value`까지 정상 요청으로 받아들일 수 있었다. 문서를 확인하면서 콜론 뒤 OWS는 허용하지만 이름과 콜론 사이 공백은 400으로 거부해야 한다는 차이를 알게 됐다. 정상 입력과 잘못된 입력을 각각 테스트해 이후 리팩토링에서도 같은 규칙을 유지할 수 있게 했다.
- 다음에도 반복하고 싶은 점:
  - HTTP parsing처럼 경계 조건이 많은 코드는 기억이나 제안에만 의존하지 않고 표준 문서, 현재 구현, 실패 테스트 순서로 확인한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - body 처리 구조를 바꾸면서 `readBody()`, `parseBodyParams()`, `parseParams()`를 따로 생각해 전체 호출 흐름을 여러 번 다시 정리했다.
- 그렇게 된 원인:
  - 수정할 메서드 하나씩 보는 데 집중했다. header를 읽은 뒤 body를 한 번 읽고, Content-Type에 따라 해석한 다음 `HttpRequest`를 만드는 전체 흐름을 먼저 확정하지 않았다.
- 다음에는 다르게 해볼 구체적인 행동:
  - parser처럼 단계가 이어지는 코드를 수정할 때는 입력부터 최종 객체 생성까지의 순서를 5줄 안팎의 의사 코드로 먼저 작성한다. 그다음 각 단계를 메서드로 나누고, InputStream처럼 한 번 소비되는 값의 소유권부터 표시한다.
