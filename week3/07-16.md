# be-DailyJob

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - `multipart/form-data` body에서 boundary를 추출하고 각 part의 header와 body를 byte 단위로 나누는 `MultipartParser`를 구현했다. `Content-Disposition`의 `name`, `filename`과 part의 `Content-Type`도 함께 파싱했다.
  - 파싱한 결과를 `MultipartPart`로 표현했다. header Map은 복사해서 저장하고 body 배열은 생성자와 getter에서 방어적으로 복사해 외부에서 내부 상태를 바꾸지 못하도록 했다.
  - `HttpRequestParser`에 `MultipartParser`를 주입하고 multipart 요청일 때만 parser를 호출하도록 연결했다. 파싱 결과는 request parameter Map에 섞지 않고 `HttpRequest`의 multipart part 목록으로 전달했다.
  - 업로드 파일을 저장하는 `FileStorage`를 만들었다. PNG와 JPEG만 허용하고 MIME type과 확장자가 일치하는지, 파일이 비어 있지 않은지, 크기 제한을 넘지 않는지 검사했다. 사용자 파일명 대신 UUID를 사용해 저장하고 정적 리소스 경로를 반환했다.
  - 게시글 작성 form을 `multipart/form-data`로 변경하고 이미지 입력을 필수로 지정했다. 본문은 없어도 되지만 이미지가 없거나 허용하지 않는 형식이면 400을 반환하도록 `ArticleController`를 수정했다.
  - 저장된 이미지 경로를 게시글에 기록하고 작성이 끝나면 생성된 게시글 ID가 포함된 메인 화면으로 redirect했다. `articles.image_path`에는 `NOT NULL` 제약을 추가하고 목업 게시글용 PNG 파일과 경로도 맞췄다.
  - multipart 정상·오류 형식, 방어적 복사, 파일 저장 정책, 실제 게시글 작성 요청을 테스트했다. 구현과 테스트를 각각 나눠 커밋하고 전체 테스트가 통과하는지 확인했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 업로드한 이미지를 메인 화면의 게시글에 렌더링하는 작업은 다음 커밋으로 남겼다.
  - 파일 저장에는 성공했지만 DB 저장에 실패했을 때 남은 파일을 제거하는 보상 처리는 아직 구현하지 않았다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - multipart 결과를 기존 request parameter Map에 넣지 않고 `List<MultipartPart>`로 따로 보관하기로 했다. `MultipartParser`는 multipart 문법 해석을 맡고, `HttpRequestParser`는 Content-Type에 따라 parser를 선택한 뒤 결과를 `HttpRequest`에 전달한다.
- 그렇게 판단한 이유:
  - form-urlencoded parameter는 문자열 이름과 값으로 충분하지만 multipart에는 filename, Content-Type, header, binary body가 함께 필요하다. 이를 모두 문자열 Map에 넣으면 파일 정보가 사라지거나 byte를 억지로 문자열로 바꿔야 한다. 컨트롤러가 원본 body를 다시 파싱하게 두면 HTTP 계층의 규칙도 컨트롤러로 새어 나온다.
- 고려했지만 선택하지 않은 대안:
  - multipart의 text part만 request parameter에 넣고 file part는 별도로 관리하는 방법도 생각했다. 하지만 한 요청의 part가 두 저장소로 나뉘어 조회 규칙이 복잡해지고 parser가 part의 용도까지 판단하게 되므로 선택하지 않았다. `MultipartPart`를 바로 record로 만드는 방법도 검토했지만 `Map`과 `byte[]`의 얕은 불변성과 배열 동등성 문제를 함께 해결해야 해서 일반 class로 유지했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - HTTP 요청 파싱, multipart 문법 해석, 파일 저장, 게시글 생성의 경계가 분명해졌다. 각 계층을 독립적으로 테스트할 수 있고, 이후 프로필 이미지 업로드를 추가해도 multipart parser를 다시 만들 필요가 없다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - multipart body를 boundary로 나눠야 했지만 이미지 byte가 포함된 전체 body를 문자열로 변환해 `split()`을 사용해도 되는지 확신하기 어려웠다. part의 시작과 끝에서 CRLF를 어디까지 포함해야 하는지도 헷갈렸다.
- 처음 추측한 원인:
  - boundary가 ASCII 문자열이므로 body 전체를 문자열로 바꾼 뒤 나눠도 된다고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - multipart 요청의 boundary, part header 구분자, closing boundary 형식을 다시 나눠 보고 byte 배열에서 필요한 위치를 찾는 `startsWith()`와 `indexOf()`의 동작 순서를 정리했다. 정상 요청뿐 아니라 boundary 누락, header 구분자 누락, 잘못된 `Content-Disposition`도 테스트했다.
- 실제 원인:
  - boundary 자체는 ASCII지만 part body는 임의의 binary 데이터다. 전체를 문자열로 디코딩하면 원본 byte가 손상될 수 있고 문자열 `split()`에는 정규식과 인코딩 문제도 따른다. 다음 boundary 앞의 CRLF와 마지막 `--`를 구분하지 않으면 파일 body에 불필요한 byte가 들어간다.
- 해결 방법:
  - boundary와 CRLF 같은 제어 문자열만 US-ASCII byte 배열로 만들고 원본 body에서 해당 배열의 위치를 직접 찾았다. part header만 ISO-8859-1 문자열로 변환하고 파일 body는 `Arrays.copyOfRange()`로 잘라 byte 배열 그대로 보관했다. 형식 오류는 `MultipartParseException`으로 알리고 HTTP 400 변환은 `HttpRequestParser`가 맡도록 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - binary protocol을 다룰 때는 전체 데이터를 문자열로 바꾸기 전에 byte 경계를 먼저 그린다. 시작 구분자, 반복 구간, 종료 구분자와 각 구간의 소유 책임을 정한 다음 정상·누락·잘못된 종료 형식 테스트부터 작성한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - multipart parser, `HttpRequest` 연결, 파일 저장, 게시글 작성 연결을 한 번에 커밋하지 않고 기능별 구현과 테스트 커밋으로 나눴다. 마지막 연결 과정에서는 실제 이미지 파일 확장자와 목업 SQL 경로를 대조하고 `image_path NOT NULL`로 인해 실패한 기존 테스트 fixture도 함께 수정했다.
- 그 접근이 도움이 된 이유:
  - 문제가 parser, 요청 객체, 저장소, controller 중 어디에서 생겼는지 커밋별로 확인할 수 있었다. PNG 파일을 추가해 놓고 SQL에는 `.jpg`를 적어 둔 불일치와, 기존 테스트가 image path에 `null`을 넣던 문제도 배포 전에 찾을 수 있었다.
- 다음에도 반복하고 싶은 점:
  - 외부 파일과 DB 경로를 함께 변경할 때는 코드만 보지 않고 실제 파일명, 스키마 제약, 목업 데이터와 테스트 fixture를 한 묶음으로 확인한다. 기능 구현과 테스트는 책임 단위로 나누고 각 단계에서 전체 테스트를 실행한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - `MultipartParser` 구현을 시작할 때 byte 배열에서 어떤 경계를 어떤 순서로 찾을지 먼저 정리하지 않아 메서드별 구현을 여러 번 질문하고 다시 맞췄다. parser 구현 커밋도 한 번에 200줄 이상 커졌다.
- 그렇게 된 원인:
  - multipart 형식을 대략 이해한 상태에서 바로 코드로 옮겼고 정상 요청 처리에 집중했다. closing boundary와 잘못된 part header 같은 실패 조건을 구현 뒤에 점검했다.
- 다음에는 다르게 해볼 구체적인 행동:
  - parser를 만들기 전 예제 요청 하나를 byte 구간별로 표시하고 각 구간의 시작·끝 조건을 표로 적는다. 가장 작은 정상 part와 boundary가 없는 실패 테스트를 먼저 만든 뒤 header 파싱과 여러 part 처리를 순서대로 확장한다.

# 주간 프롬프트 회고 템플릿

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:

> boundary 를 parser가 몰라도 되게 하는게 나으려나? 책임 분리 상
>
> 아 내 말은 requestParser가 boundary를 추출할 책임까지는 없지 않나 싶은 거였음 contentType을 넘겨주면 될 거 같아서

선택한 이유:

- 단순히 동작하는 코드를 묻지 않고 boundary 추출 책임이 어느 객체에 있어야 하는지 질문했다.
- 처음 표현이 모호하다는 것을 알아차린 뒤 `HttpRequestParser`와 `MultipartParser`를 구분해 의도를 다시 설명했다.
- `Content-Type` 전체를 넘기는 대안까지 제시해 책임, 응집도, 결합도를 기준으로 설계를 비교할 수 있었다.

### 2. 원본 프롬프트 구조 분석
- 문제 상황:
  - multipart 요청을 처리하려면 `Content-Type`에서 boundary를 꺼내야 했다. 이 작업을 `HttpRequestParser`가 맡을지, 실제 multipart 문법을 해석하는 `MultipartParser`가 맡을지 결정해야 했다.
- 원하는 결과를 어떻게 요구했는가:
  - 두 parser의 책임을 비교하고 `HttpRequestParser`가 boundary 추출까지 아는 구조가 적절한지 검토해 달라고 요청했다.
- 필요한 맥락이 포함되어 있는가:
  - boundary가 `Content-Type`의 parameter이고 현재 `HttpRequestParser`가 `MultipartParser`를 호출한다는 직전 대화의 맥락이 있었다. 두 클래스의 현재 method signature까지 적었다면 프롬프트만 따로 읽어도 이해하기 쉬웠을 것이다.
- 판단 기준이나 제약 조건이 있는가:
  - 책임 분리를 판단 기준으로 제시했다. 다만 응집도, 결합도, 테스트 편의성 중 무엇을 더 중요하게 볼지는 명시하지 않았다.
- 출력 방식이 명확한가:
  - 특정 형식은 요구하지 않았지만 두 대안 중 어떤 구조가 더 나은지와 그 이유를 설명하는 답을 기대한다는 점은 드러난다.
- 나의 가설, 우려, 의도가 드러나는가:
  - `HttpRequestParser`가 multipart 내부 규칙인 boundary 추출까지 맡으면 책임이 커진다는 우려가 드러난다. 대신 `Content-Type` 전체를 `MultipartParser`에 넘기자는 가설도 포함되어 있다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
현재 `HttpRequestParser`가 요청의 `Content-Type`을 확인한 뒤 `MultipartParser`를 호출하는 구조야.
multipart parsing에는 `Content-Type` parameter인 boundary가 필요한데, 어느 클래스가 boundary를 추출해야 할지 고민하고 있어.

두 가지 안을 비교해줘.

1. `HttpRequestParser`가 boundary를 추출하고 `MultipartParser.parse(body, boundary)`를 호출한다.
2. `HttpRequestParser`는 `Content-Type` 전체를 넘기고 `MultipartParser.parse(body, contentType)`가 boundary를 추출한다.

HTTP 요청 parser와 multipart parser의 책임, 응집도, 결합도, 단위 테스트 범위를 기준으로 비교해줘.
현재는 `multipart/form-data`만 지원하며 범용 body parser 같은 추가 추상화는 만들지 않을 생각이야.
더 적절한 안을 하나 추천하고, 추천한 구조의 method signature와 각 클래스가 맡을 일을 짧게 보여줘.
```

### 4. AI가 개선한 프롬프트 분석
- 어느 클래스가 boundary를 추출해야 하는지 두 대안을 같은 조건에서 비교할 수 있게 정리했다.
- 책임 분리라는 넓은 표현을 응집도, 결합도, 테스트 범위라는 구체적인 판단 기준으로 풀었다.
- 현재 지원 범위와 추가 추상화를 만들지 않겠다는 제약을 넣어 과한 설계 제안을 막았다.
- 추천만 요구하지 않고 method signature와 각 클래스의 역할도 요청해 답을 바로 코드에 반영할 수 있게 했다.
