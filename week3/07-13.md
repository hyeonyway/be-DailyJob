> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - `400`, `404`, `405`, `500` HTML 오류 페이지를 만들고, 요청 파싱·라우팅·컨트롤러 예외 흐름에 연결했다.
  - 존재하지 않는 경로와 허용하지 않는 HTTP 메서드를 구분해 `404`와 `405`를 각각 반환하도록 수정하고 테스트했다.
  - H2 초기화, `schema.sql` 실행, `UserRepository`/`JdbcUserRepository`를 추가했다. `UserService`, `AuthService`는 repository를 주입받도록 바꾸고 기존 static `Database`는 제거했다.
  - `Article` 모델과 `articles` schema를 추가했다. JDBC 게시글 저장소에서 저장, 최신·단건·이전·다음 글 조회, 좋아요 수 증가를 구현했다.
  - H2 기반 사용자·게시글 저장소 테스트와 요청 흐름 회귀 테스트를 작성했다.
  - 현재 구현 상태에 맞춰 Step7 전체 계획과 PR별 세부 계획을 다시 정리했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 게시글을 실제 요청 흐름에 연결하는 `ArticleService`, `ArticleController`, 메인 화면 렌더링은 다음 작업으로 남아 있다.
  - 이미지 업로드 규칙은 아직 구현하지 않았으므로 `image_path`의 `NOT NULL` 제약은 PR3에서 적용한다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 사용자 저장소를 static `Database`에서 H2 기반 `UserRepository`로 전환하고, 게시글도 같은 JDBC repository 구조를 따르기로 했다.
- 그렇게 판단한 이유:
  - 회원가입과 로그인은 이미 저장 방식에 따라 동작해야 하는 기능이었다. service가 static 저장소를 직접 알면 게시글·댓글까지 DB를 확장할 때 의존성이 계속 퍼진다. repository를 경계로 두면 저장 방식과 도메인 흐름을 분리할 수 있다.
- 고려했지만 선택하지 않은 대안:
  - JDBC 반복 코드를 줄이려고 직접 `JdbcTemplate`과 비슷한 추상화를 만드는 방법을 생각했다. 지금은 repository 수가 많지 않고 raw JDBC를 익히는 단계이므로, `try-with-resources`와 `ResultSet` 매핑을 직접 유지하기로 했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - H2를 사용하는 테스트에서 실제 저장·조회·중복 확인을 검증할 수 있게 됐다. 이후 `ArticleRepository`, `CommentRepository`도 같은 경계를 따라 추가할 수 있다.

### 3. 가장 막혔던 문제와 해결 과정
- 오늘은 작업을 멈출 정도로 막힌 문제는 없었다. repository의 SQL과 반환 타입을 정하는 과정에서 확인할 내용은 있었지만, 요구사항과 JDBC 동작을 하나씩 대조하며 해결했다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 게시글 repository를 구현하면서 SQL과 모델 매핑을 바로 확정하지 않고, `id` 생성 주체, 이전·다음 글 조회 기준, `Optional` 사용 범위, 좋아요 증가 결과를 하나씩 확인했다. 구현 뒤에는 저장·조회·이전/다음·좋아요를 실제 H2 테스트로 고정했다.
- 그 접근이 도움이 된 이유:
  - 이전·다음 글이 없는 상태는 정상 흐름이고, 없는 게시글 id는 404로 처리해야 한다는 차이를 코드 계약에 반영할 수 있었다. `increaseLikeCount()`도 변경 행 수를 반환하게 해 없는 게시글을 구분할 기반을 남겼다.
- 다음에도 반복하고 싶은 점:
  - repository 메서드를 만들기 전에 입력값, 결과가 없을 때의 반환값, 호출자가 판단해야 할 상태를 먼저 정하고 테스트까지 같은 단위로 작성한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - `JdbcArticleRepository`를 처음 작성할 때 조회 SQL의 `like_count` 누락, 이전·다음 조회 parameter 바인딩 누락, `Optional` 메서드의 `null` 반환처럼 매핑과 반환 계약이 어긋난 부분이 있었다.
- 그렇게 된 원인:
  - repository 메서드별 SQL 목록과 `ResultSet -> Article` 매핑 필드를 먼저 대조하지 않고 구현을 진행했다. 생성 id 반환 방식도 구현 도중에 결정했다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 구현 전에 repository 메서드마다 SQL, parameter, 반환 타입, 없는 결과의 처리 방식을 짧게 적는다. 조회 SQL에는 모델 매핑에 필요한 모든 column이 포함됐는지 `createArticle()`과 함께 확인한다.
